# Requirements: Model Failure Lab

**Defined:** 2026-04-02
**Core Value:** Make structured LLM failure analysis simple, reproducible, and easy to inspect from local artifacts.

## v3.0 Requirements

### Packaged Distribution

- [x] **DX-02**: User can install and run `failure-lab` cleanly from a standard package install without relying on editable local development setup.
- [x] **DX-03**: User can follow one package-install quickstart to run `demo`, `run`, `report`, and `compare` with bundled assets and inspect the resulting local artifacts.

### Ollama Adapter

- [x] **ADAPT-01**: User can run the same dataset-to-report loop through an Ollama adapter that matches the existing artifact contract.
- [x] **ADAPT-02**: User can configure Ollama model selection and connection settings from the CLI without patching code.
- [x] **ADAPT-03**: User can inspect Ollama-backed runs, reports, and comparisons in the existing artifact-backed debugger without adapter-specific UI branches.

## v4 Requirements

### Future Distribution And Adapters

- **DX-04**: User can install only the optional provider extras they need instead of pulling every adapter dependency.
- **ADAPT-04**: User can run additional local or hosted adapter families through the same adapter registry and artifact contract.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Additional new adapter families beyond Ollama | Prove one broader adapter path first before widening provider breadth again |
| Another debugger interaction redesign | `v2.1` already delivered the debugger continuity pass; this milestone is about reach, not UI reinvention |
| Hosted artifact services, databases, or remote execution infrastructure | Local JSON artifacts remain the canonical product contract |
| Learned classifiers or benchmark-architecture revival | Keep the milestone focused on distribution and adapter breadth over engine-semantics churn |
